# Store Backend API
REST API для интернет-магазина с ролевой моделью доступа.

## Stack
- Python
- FastAPI 
- PostgreSQL 
- SQLAlchemy Async
- Alembic
- JWT Authentication
- bcrypt

## Как запустить проект
1. Клонировать репозиторий:
git clone https://github.com/IliyMix1/job_task
2. Установить зависимости:
pip install -r requirements.txt
3. Создать .env на основе .env.example
4. Применить миграции(alembic upgrade head)
5. Заполнить БД тестовыми данными, запустив специальный скрипт(python seed.py)
6. Запустить сервер:
uvicorn main:app --reload

## Тестовые сценарии
1. Зарегистрировать пользователя через /auth/reg
2. Войти через /auth/login
3. Скопировать access_token
4. Передавать токен в Swagger через кнопку Authorize
5. роверить доступ к /my/profile
6. Проверить logout через /auth/logout
7. Проверить, что после logout старый токен больше не проходит авторизацию
8. Под администратором проверить доступ к /admin/users, /admin/access_rules и /products

## Основной функционал
- Регистрация пользователя
- Логин по email и паролю
- Logout c отзывом JWT-токена через чёрный список
- Просмотр, обновление и мягкое удаление(is_active=False) своего профиля
- Ролевая система доступа
- Проверка прав через таблицу access_rules
- CRUD для управления пользователями со стороны администратора
- CRUD для управления правилами доступа со стороны администратора
- CRUD для бизнес-ресурса products

## Система аутенфикации
После регистрации пароль пользователя проходит через хэширование с помощью bcrypt, и уже потом сохраняется в базу данных.

При логине пользователь передаёт email и пароль. Если данные указаны корректно, сервер выдаёт JWT-токен. В токене хранится id пользователя, уникальный id токена(jti) и срок действия токена(exp).

После обращения к защищённым эндпоинтам клиент обязан передавать токен в заголовке: Authorization: Bearer <access_token>
Если токен отсутствует, некорректен, истёк или был отозван через logout - возвращается ошибка 401 Unauthorized

## Logout
Сервер сам по себе не хранит информацию о сессии пользователя(JWT-токен - stateless), поэтому для реализации logout была создана таблица token_blacklist.

При выходе из аккаунт jti текущего токена сохраняется в таблицу token_blacklist. При каждом запросе к защищённому эндпоинту сервер проверяет, есть ли jti токена в blacklist. Если токен найден - пользователь считается неавторизованным

## "Мягкое" удаление
При удалении аккаунта, запись пользователя фактически не удаляется из базы данных. Вместо этого, значение поля is_active меняется на False.

Пользователь, у которого is_active=False, больше не может войти в систему и использовать защищённые эндпоинты, однако запись сохраняется в базе данных

## Система авторизации
Авторизация построена на связке трёх сущностей: roles, resources и access_rules.
roles - таблица всех ролей(примеры ролей: user, admin, manager). Каждый пользователь имеет одну роль с уникальным role_id

resources - таблица ресурсов приложения, к которым можно ограничивать доступ(В проекте используются: profile, users, products, access_rules)

access_rules - таблица правил доступа. Она связывает роль и ресурс, а также хранит набор bool-разрешений. Поля:
- role_id
- resource_id
- can_create
- can_read_own
- can_read_all
- can_update_own
- can_update_all
- can_delete_own
- can_delete_all
Primary key является составным: role_id + resource_id

Примеры логики:
- Пользователь с ролью user может читать, обновлять и удалять свой профиль
- Пользователь с ролью admin может управлять пользователями
- Пользователь с ролью admin может читать, создавать, изменять и удалять правила доступа
- Если для роли и ресурса нет правила, доступ запрещён
- Если правило есть, но нужное разрешение равно False, доступ запрещён

## Проверка прав
Для проверки прав реализован класс PermissionChecker. Он получает:
* Название ресурса
* Название конкретного разрешения
* Текущего пользователя из JWT-токена
* Роль пользователя
* Правило доступа из таблицы access_rules

Если пользователь не определён, возвращается ошибка 401 Unauthorized.
Если пользователь определён, но у него нет доступа к ресурсу, возвращается ошибка 403 Forbidden

## Основные эндпоинты
### Auth
POST /auth/reg
POST /auth/login
POST /auth/logout
### Profile
GET /my/profile
PATCH /my/profile
DELETE /my/profile
### Admin users
GET /admin/users
PATCH /admin/users/{user_id}
DELETE /admin/users/{user_id}
### Admin access rules
GET /admin/access_rules
POST /admin/access_rules
PATCH /admin/access_rules/{role_id}/{resource_id}
DELETE /admin/access_rules/{role_id}/{resource_id}
### Products
GET /products
POST /products
PATCH /products/{product_id}
DELETE /products/{product_id}


## Принцип работы ошибок
- 401 Unauthorized - пользователь не авторизован, токен отсутстсвует, некорректен или занесён в blacklist
- 403 Forbidden - пользователь авторизован, но его роль не обладает достаточными правами доступа
- 404 Not Found - запрашиваемая запись не найдена
- 409 Conflict - конфликт данных(например, email уже занят)
